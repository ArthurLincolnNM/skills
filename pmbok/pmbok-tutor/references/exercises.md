## KA 1: INTEGRATION MANAGEMENT
### MC001
**Q:** Você é PM de um projeto de software. Na reunião semanal, um stakeholder pede uma mudança no escopo. Qual é a PRIMEIRA coisa que você deve fazer?
**A)** A) Implementar a mudança imediatamente para manter o stakeholder feliz
**B)** B) Avaliar o impacto da mudança através do Perform Integrated Change Control
**C)** C) Pedir aprovação do comitê de controle de mudanças
**D)** D) Recusar porque mudanças de escopo são proibidas
**Correct:** B — Perform Integrated Change Control é o processo para avaliar TODAS mudanças propostas. Não se implementa nem recusa sem análise.
**PMBOK:** Integration Management > M&CG > 4.5 Perform Integrated Change Control
**Why B > C:** CCB pode aprovar, mas primeiro o PM deve analisar e submeter. O PM não delega análise.
### MC002
**Q:** Qual é o principal resultado do processo Develop Project Management Plan?
**A)** A) Project Charter
**B)** B) Scope Baseline
**C)** C) Project Management Plan (integração de todos os planos subsidiários)
**D)** D) WBS
**Correct:** C — O Project Management Plan é o plano maestro que integra todos os management plans (scope, schedule, cost, quality, etc).
**PMBOK:** Integration > Planning > 4.2 Develop Project Management Plan
### MC003
**Q:** Você está fechando um projeto. Qual das seguintes atividades NÃO é parte do Close Project or Phase?
**A)** A) Coletar lições aprendidas
**B)** B) Arquivar todos os documentos do projeto
**C)** C) Liberar os recursos do projeto
**D)** D) Obter aprovação do sponsor para iniciar o projeto
**Correct:** D — Obter aprovação do sponsor para iniciar é do Initiating (Develop Project Charter). Close é sobre FINALIZAR.
**PMBOK:** Integration > Closing > 4.7 Close Project or Phase
### VF001
**Q:** "O Project Management Plan é desenvolvido uma única vez e nunca é alterado durante o projeto."
**A)** True
**B)** False
**Correct:** False — O PMP é um documento "living". É atualizado através do Perform Integrated Change Control quando mudanças approved são incorporadas. É baseline, não imutável.
**PMBOK:** Integration > Planning > 4.2
### VF002
**Q:** "O PM é sempre o accountable por todos os aspectos do projeto, mesmo que não seja diretamente responsável por alguns."
**A)** True

**Correct:** True — Accountability no PMI framework significa que o PM tem responsabilidade final mesmo quando delega execução.
**PMBOK:** Integration > General

## KA 2: SCOPE MANAGEMENT
### MC004
**Q:** Qual é a MELHOR definição de escopo do projeto?
**A)** A) O orçamento total do projeto
**B)** B) O prazo final do projeto
**C)** C) A soma de todos os produtos, serviços e resultados do projeto
**D)** D) O número de atividades a serem executadas
**Correct:** C — Escopo do projeto = todo o trabalho necessário para entregar o produto/serviço. É diferente do schedule (quando) e do budget (quanto).
**PMBOK:** Scope > Planning > 5.3 Define Scope
### MC005
**Q:** Você está criando a WBS. Qual é a regra fundamental?
**A)** A) 100% rule — incluir todo o trabalho necessário
**B)** B) 80/20 rule — focar no trabalho mais importante
**C)** C) 50/50 rule — separar trabalho interno e externo
**D)** D) Prioridade do sponsor
**Correct:** A — 100% rule significa que a WBS deve incluir todo o trabalho (e ONLY that work).
**PMBOK:** Scope > Planning > 5.4 Create WBS
### MC006
**Q:** Qual é a DIFERENÇA entre Validate Scope e Control Scope?
**A)** A) São iguais, nomes diferentes
**B)** B) Validate Scope = aceitação formal pelo stakeholder. Control Scope = monitorar mudanças.
**C)** C) Validate Scope = interno. Control Scope = externo.
**D)** D) Nenhuma das anteriores.
**Correct:** B — Validate Scope é sobre ACEITAR formalmente o deliverable. Control Scope é sobre MONITORAR mudanças ao escopo.
**PMBOK:** Scope > Executing > 5.5 Validate Scope; Scope > M&CG > 5.6 Control Scope
### VF003
**Q:** "Requirements e Scope são a mesma coisa."
**A)** True
**B)** False
**Correct:** False — Requirements = features/funções QUE o projeto precisa TER. Scope = todo o trabalho PARA ENTREGAR essas features.
**PMBOK:** Scope > Planning > 5.2, 5.3

## KA 3: SCHEDULE MANAGEMENT
### MC007
**Q:** Você calculou o caminho crítico de um projeto com 4 caminhos possíveis: CP1=45 dias, CP2=50 dias, CP3=52 dias, CP4=48 dias. Qual é a duração do projeto?
**A)** A) 45 dias
**B)** B) 50 dias
**C)** C) 52 dias (longest path = minimum duration)
**D)** D) 48 dias
**Correct:** C — O caminho crítico É o longest path. O projeto NÃO pode ser completado mais rápido que o CP. Duração mínima = 52 dias.
**PMBOK:** Schedule > Planning > 6.6 Develop Schedule
### MC008
**Q:** Uma atividade tem Early Start=10, Early Finish=18, Late Start=12, Late Finish=20. Qual é o Total Float dessa atividade?
**A)** A) 0
**B)** B) 2 (Total Float = LS - ES = 12-10 = 2)
**C)** C) 4
**D)** D) 8
**Correct:** B — Total Float = LS - ES = Late Start - Early Start = 12 - 10 = 2 dias. Activities com float > 0 podem ser atrasadas sem afetar o prazo do projeto.
**PMBOK:** Schedule > Planning > 6.6 Develop Schedule
### MC009
**Q:** Você está fazendo análise quantitativa de cronograma. Uma atividade tem O=3, M=5, P=13. Qual é a estimativa PERT?
**A)** A) 3
**B)** B) 5
**C)** C) 6
**D)** D) 13
**Correct:** C — PERT = (O + 4M + P) / 6 = (3 + 20 + 13) / 6 = 36 / 6 = 6 dias.
**PMBOK:** Schedule > Planning > 6.6
### VF004
**Q:** "Você pode comprimir uma atividade que NÃO está no caminho crítico para encurtar a duração do projeto."
**A)** True
**B)** False
**Correct:** False — Comprimir atividades fora do CP não encurta a duração do projeto (CP é o longest path). Só atividades NO CP afetam o prazo final.
**PMBOK:** Schedule > Planning > 6.6
### VF005
**Q:** "Crashing é a técnica de adicionar recursos a uma atividade para encurtar sua duração."
**A)** True
**B)** False
**Correct:** True — Crashing adiciona recursos (ou compressão) custando mais para encurtar duração. Fast tracking é sobre paralelizar atividades.
**PMBOK:** Schedule > Planning > 6.6

## KA 4: COST MANAGEMENT
### MC010
**Q:** Um projeto tem BAC = $100,000. O % completo é 40% mas o custo real é $50,000. Qual é o CPI? O que isso significa?
**A)** A) CPI = 0.8 — projeto está over budget
**B)** B) CPI = 0.8 — projeto está behind schedule
**C)** C) CPI = 1.2 — projeto está under budget
**D)** D) CPI = 1.0 — projeto está on budget
**Correct:** A — EV = 40% × $100,000 = $40,000. CPI = EV / AC = $40,000 / $50,000 = 0.80. CPI < 1 = over budget.
**PMBOK:** Cost > M&CG > 7.4 Control Costs
**Interpretation:** O projeto gastou $50k para fazer apenas $40k de trabalho. Custo real excedeu valor agregado.
### MC011
**Q:** Um projeto tem SPI=0.9 e CPI=0.9. O que está acontecendo?
**A)** A) Projeto está ahead of schedule e under budget
**B)** B) Projeto está behind schedule e over budget
**C)** C) Projeto está on track
**D)** D) Projeto está ahead of schedule mas over budget
**Correct:** B — SPI < 1 → behind schedule. CPI < 1 → over budget. Ambos < 1 = projeto com problemas de prazo E custo.
**PMBOK:** Cost > M&CG > 7.4; Schedule > M&CG > 6.7
### MC012
**Q:** Um projeto tem BAC = $200,000, AC = $100,000, EV = $80,000. Qual é o EAC mais provável?
**A)** A) $200,000
**B)** B) $160,000
**C)** C) $250,000 (EAC = BAC/CPI = 200000/(80000/100000) = 200000/0.8 = 250000)
**D)** D) $180,000
**Correct:** C — EAC = BAC / CPI = $200,000 / 0.80 = $250,000. Assuming cost performance rate continues.
**PMBOK:** Cost > M&CG > 7.4
### VF006
**Q:** "EV (Earned Value) é sempre medido em horas para projetos de serviço."
**A)** True
**B)** False
**Correct:** False — EV é medido na mesma unidade do BAC. Se BAC é em $, EV é em $. Se BAC é em horas, EV é em horas.
**PMBOK:** Cost > M&CG > 7.4
### VF007
**Q:** "CPI < 1 indica que o projeto está GASTANDO mais do que deveria para o trabalho realizado."
**A)** True
**B)** False
**Correct:** True — CPI = EV/AC. Quando AC > EV, o denominador é maior → CPI < 1.
**PMBOK:** Cost > M&CG > 7.4

## KA 5: QUALITY MANAGEMENT
### MC013
**Q:** Qual é a DIFERENÇA entre Quality Assurance e Quality Control?
**A)** A) São iguais
**B)** B) QA = melhoria de PROCESSOS (prevenção). QC = validação de OUTPUTS (inspeção).
**C)** C) QA = inspeção. QC = prevenção.

## KA 6: RESOURCE MANAGEMENT
### MC014
**Q:** Um projeto tem 8 membros de equipe. Quantos canais de comunicação existem?
**A)** A) 8
**B)** B) 16
**C)** C) 28 (channels = n(n-1)/2 = 8×7/2 = 28)
**D)** D) 64
**Correct:** C — Channels = 8 × (8 - 1) / 2 = 8 × 7 / 2 = 28. Communication overload é um problema real em projetos grandes.
**PMBOK:** Resource > Planning > 9.1; Communications > Planning > 10.1
### MC015
**Q:** Qual é a ordem correta dos estágios de desenvolvimento de equipe de Tuckman?
**A)** A) Forming → Storming → Norming → Performing → Adjourning
**B)** B) Forming → Storming → Norming → Performing
**C)** C) Norming → Storming → Forming → Performing
**D)** D) Performing → Norming → Storming → Adjourning
**Correct:** A — Tuckman: Forming → Storming → Norming → Performing → Adjourning. O Adjourning está presente no PMBOK 6 (seção 9.4) e representa a dissolução da equipe ao fim do projeto.
**PMBOK:** Resource > Executing > 9.4 Develop Team
### VF009
**Q:** "Teoria Y assume que pessoas precisam de controle externo e incentivos para performar."
**A)** True
**B)** False
**Correct:** False — Teoria Y = pessoas são auto-motivadas e auto-direcionadas. Teoria X = controle externo.
**PMBOK:** Resource > Planning > 9.1

## KA 8: RISK MANAGEMENT
### MC016
**Q:** Você identificou um risco com Probabilidade = 0.3 e Impacto = $40,000. Qual é o EMV?
**A)** A) $0.3
**B)** B) $40,000
**C)** C) $12,000 (EMV = 0.3 × $40,000 = $12,000)
**D)** D) $133,333
**Correct:** C — EMV = Probability × Impact = 0.3 × $40,000 = $12,000. Used em decision trees.
**PMBOK:** Risk > Planning > 11.2 Identify Risks
### MC017
**Q:** Qual é a MELHOR estratégia para um risco de AMEAÇA (threat) com alta probabilidade e alto impacto?
**A)** A) Accept — aceitar e lidar se acontecer
**B)** B) Avoid — eliminar completamente a causa
**C)** C) Mitigate — reduzir probabilidade/impacto
**D)** D) Transfer — transferir para terceiros
**Correct:** B — Para ameaças HIGH PROBABILITY + HIGH IMPACT, Avoid é melhor porque elimina completamente. Mitigate reduz mas não elimina. Transfer transfere o impacto mas não remove. Accept é para riscos residuais ou de baixo impacto.
**PMBOK:** Risk > Planning > 11.5 Plan Risk Responses
### MC018
**Q:** Qual é a DIFERENÇA entre fallback plan e workaround?
**A)** A) São iguais
**B)** B) Fallback = plano alternativo quando o primary fails. Workaround = resposta para risco NÃO planejado.
**C)** C) Workaround = primary. Fallback = secondary.
**D)** D) Nenhuma das anteriores.
**Correct:** B — Fallback plans são PLANEJADOS (para quando primary plan falha). Workarounds são respostas IMPROVISADAS para riscos que não tinham plano pré-definido.
**PMBOK:** Risk > Planning > 11.5
### VF011
**Q:** "Um workaround é uma estratégia de resposta a risco que foi pré-planejada."
**A)** True
**B)** False
**Correct:** False — Workaround = resposta IMPROVISADA para um risco que NÃO tinha plano. Contingency plans são pré-planejados. Fallback plans são pré-planejados para quando contingency fails.
**PMBOK:** Risk > Planning > 11.5

## KA 9: PROCUREMENT MANAGEMENT
### MC019
**Q:** Um projeto tem escopo bem definido e entregas fixas. Qual tipo de contrato é mais apropriado?
**A)** A) Cost Plus Percentage of Cost (CPPC)
**B)** B) Firm Fixed Price (FFP)
**C)** C) Time and Materials (T&M)
**D)** D) Cost Plus Incentive Fee (CPIF)
**Correct:** B — FFP é melhor quando o escopo é WELL-DEFINED (baixa incerteza). Seller assume risco. Buyer pays FIXED price.
**PMBOK:** Procurement > Planning > 12.1 Plan Procurement Management
### VF012
**Q:** "CPPC (Cost Plus Percentage of Cost) é o tipo de contrato mais usado em projetos modernos."
**A)** True
**B)** False
**Correct:** False — CPPC é OBSOLETO. Incentiva o vendedor a AUMENTAR custos para aumentar fee. É proibido ou desencorajado em muitos contextos.
**PMBOK:** Procurement > Planning > 12.1

## KA 10: STAKEHOLDER MANAGEMENT
### MC020
**Q:** Você está criando uma matriz de engajamento de stakeholders. Um stakeholder está Current=C e Desired=D. O que você deve fazer?
**A)** A) Ignorar — não é problema seu

**C)** C) Reduzir o nível de engajamento para C

**Correct:** B — Quando Desired > Current, o stakeholder precisa ser mais envolvido. Manage Stakeholder Engagement = closing that gap.
**PMBOK:** Stakeholder > Executing > 13.3 Manage Stakeholder Engagement

**B)** False
